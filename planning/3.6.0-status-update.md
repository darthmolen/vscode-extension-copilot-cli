## Feature Summary

Full CRUD UI for named custom agents with scoped tools and system prompts. Three non-deletable built-in agents (Planner, Implementer, Reviewer) ship out of the box. Users can create, edit, and delete their own agents. Agents are injected into every SDK session at creation time. Per-message agent selection via `@agentName` mention prefix. Sticky session agent via `/agent <name>` slash command.

## Known Limitations / Follow-up Backlog

- **Agent hot-reload**: Agents are snapshotted at session creation. Changing agents requires starting a new session.
- **Tools field**: Free-text comma-separated for v1. A constrained multi-select picker with known tool names is deferred.
- **Active agent indicator**: The toolbar badge shows the sticky agent but there's no per-message indicator in the chat transcript showing which agent responded.
- **Agent reset button**: No "reset to defaults" for built-in agents that have been overridden. Users delete the override manually from workspace settings.
- **/agent with no args**: Clears sticky agent (passes empty string). Not explicitly documented in the slash command panel description yet.

## Code Review

**Reviewed:** 2026-03-15
**Reviewer:** Claude Code (code-reviewer subagent)
**Commit:** `e0610e9` â€” 37 files, ~2586 insertions
**Verdict:** Ship with fixes

### Critical Issues

**1. XSS vulnerability in CustomAgentsPanel.js**
`_renderList()` injects agent data directly into `innerHTML` without sanitization â€” `agent.displayName`, `agent.name`, `agent.description` all interpolated raw. Similarly, `_showForm()` puts values into `value="${name}"` template literals. A malicious `.vscode/settings.json` in a cloned repo could inject arbitrary HTML. The codebase already has `escapeHtml()` in `app/utils/webview-utils.js` but it's not used here.

**Fix:** Import and use `escapeHtml()` for all agent fields rendered into HTML. For form fields, set `.value` programmatically after element creation rather than interpolating into templates.

### Important Issues

**2. SelectAgentPayload misplaced in messages.ts** â€” Defined in the "Extension â†’ Webview" section but it's actually a Webview â†’ Extension message (correctly wired in `WebviewMessageType` union). Misleading placement will confuse maintainers. Move it to the Webview â†’ Extension section.

**3. Agent name validation vs @mention regex mismatch** â€” `CustomAgentsService.save()` only validates non-empty/non-whitespace. `parseAgentMention()` uses `/^@([a-z0-9_-]+)\s*/` which silently fails on uppercase, spaces, or special chars. A user creating "My Agent" through the UI can never `@mention` it. Add slug format validation (`/^[a-z0-9_-]+$/`) in the service, or document the requirement in the form placeholder.

**4. `config.update(key, value, true)` saves globally, not per-workspace** â€” The `true` third arg means `ConfigurationTarget.Global`. Agents persist across all workspaces. If intentional, add a comment explaining why. If not, use `ConfigurationTarget.Workspace`.

**5. Empty prompt allowed** â€” Form falls through to `''` for prompt; service doesn't validate. An empty-prompt agent may cause silent SDK failures. Require non-empty prompt or warn.

**6. Agent select/deselect race condition** â€” `sendMessage()` calls `agent.select()` before `sendAndWait()` and `agent.deselect()` in `finally`. Rapid consecutive messages with different agents could race. No test covers this. Worth a comment at minimum.

**7. Fragile event args wiring** â€” `main.js` line 111: `eventBus.on('selectAgent', (args) => rpc.selectAgent(args && args[0] ? args[0] : ''))` depends on EventBus passing args array as first param. Correct but fragile â€” consider destructuring.

### Minor Issues

**8. Two separate CustomAgentsService instances** â€” `ChatViewProvider` and `SDKSessionManager` each create their own. Functionally correct (reads config each call), but prevents shared caching. Consider passing a single instance from extension.ts.

**9. Dead import in main.js** â€” `StatusBar` imported (line 10) but never instantiated (commented out lines 70-72).

**10. Source-code-level test assertions** â€” `custom-agents-sdk-wiring.test.js` and `slash-command-agent.test.js` read `.ts` source with `fs.readFileSync` and assert on string contents. Brittle to renames/refactoring. Replace with behavioral tests eventually.

**11. Missing JSDoc** â€” `onActiveAgentChanged` in `WebviewRpcClient.js` (line 569) lacks the JSDoc block every other handler has.

### Strengths

1. **Clean RPC contract** â€” All 6 message types properly defined with payloads, registered in both unions and type guards. Extension and webview sides updated in lockstep.
2. **Build system correct** â€” esbuild.js has the required dist dir variable, mkdirSync, and copyFileSync per CLAUDE.md rules.
3. **Thorough test coverage** â€” ~70 new tests across 6 new + 5 modified test files. Tests use real production code, not mocks.
4. **Clean separation of concerns** â€” Service handles config CRUD, ChatViewProvider handles RPC routing, SDKSessionManager handles SDK, CustomAgentsPanel handles UI.
5. **Defensive agent select/deselect** â€” `finally` block ensures deselect always runs; errors caught and logged.
6. **Built-in agent protection** â€” Service refuses deletion; UI hides delete button for `builtIn: true`. Flag stripped before SDK.
7. **Full suite passes** â€” 1388 passing, only 3 pre-existing baseline failures.

### Summary

Well-structured feature that follows established patterns (RPC contract, component hierarchy, service layer, esbuild copy). The XSS vulnerability in `CustomAgentsPanel` rendering user-configurable data via `innerHTML` without escaping is the one must-fix before release. The agent name validation gap will cause user confusion when names don't match the `@mention` regex. Remaining issues are addressable in follow-up.
